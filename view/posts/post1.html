Exercicios para aprender de forma prática... BDD e TDD

Posted by M@lu on March 22, 2017

A proposta para este post era fazer uma reflexão envolvendo os assuntos: Testes (BDD e TDD), Padrões de Projeto e Codificação. Qual a melhor forma de envolver os três? É sempre necessário que os três andem juntos? Essas coisas...

Bem, como uma visão geral do desenvolvimento, eu definiria como complicado. Complicado por vários motivos, mas o principal... por causa da lógica.

Para que um código fique bem escrito e bem estruturado o primeiro passo, na minha opinião, é que a lógica do negócio tenha sido bem compreendida. A mas porque? Sempre dá pra modificar depois. Sim, mas a maneira ideal de se desenvolver um programa é definindo os testes para os comportamentos, para que em seguida possam ser criadas as classes e os métodos baseados nos testes. Então vamos lá comigo, se a lógica tiver confusa, logo, os testes estarão confusos o que resultará em... código ruim.

Por isso que classifiquei a experiência como complicada. Primeiro porque nem sempre é tão simples de compreender a lógica do negócio, e segundo porque além de ter essas etapas bem definidas era necessário unir ao desenvolvimento os padrões de projeto.

Padrões de Projeto, quando utilizados de forma correta, colaboram para a obtenção de um design flexível, mais coeso e menos acoplado. O que proporciona melhor manutenabilidade, flexibilidade, reuso de código e aumento na qualidade.

Os padrões utilizados ao longo dos exercicios foram: Observer, Strategy, Interpreter, Chain of Responsability e MVC.

Meu contato com o estudo dos padrões de projeto foi bem breve durante a disciplina de Programação Orientada a Objetos 2 à 1 ano atrás, portanto tive uma certa dificuldade na implementação dos padrões. A compreensão sobre como cada padrão deve ser utilizado não é tão complexo, mas de compreender até aplicar ao seu projeto tem uma certa distância.

O padrão que eu tive bem mais contato é o padrão MVC. Eu costumo programar mais para front-end, e nos frameworks que utilizo, esse padrão, ou uma variância dele, é normalmente utilizado. Então a estruturação do projeto dentro do padrão MVC não foi tão complexo para mim.

Então... Respondendo as primeiras perguntas que fiz. A melhor maneira de combinar os três seria BDD, TDD, Codigo, Refatoração, Padrões de Projeto, Testes. Nem sempre será possivel, ou necessário, associar padrões de projetos ao seu programa, mas é sempre indicado que testes e codificação andem sempre juntos.
